# Smile-Chat 实验功能进一步开发说明（Draft）

> 目标：在现有 Smile-Chat（启明）聊天系统基础上，补齐“实验流程”相关的入口页/结束页/状态记录，并确保后端实验分组（`condition`）与提示词切换逻辑清晰可控。

---

## 0. 现状核对（基于当前仓库）

- **后端已具备**
  - **三组提示词**：`backend/prompts/{情感表露,事实表露,无表露}.txt`
  - **按 condition 选 prompt**：`PromptManager.get_system_prompt(condition)`
  - **v2 协议化 chat**：`AIService.chat_with_context_v2()` 返回 `reply/segments/meta`，并记录 `did_self_disclosure`、`relationship_stage` 等。
  - **condition 获取逻辑**：`SessionStateService.get_condition(user_id)` 优先读取用户数据字段 `self_disclosure_condition`。

- **前端已具备**
  - `/chat` 页面内置 **16min 计时**（15min 提示 + 16min 强制结束遮罩），并按天存到 `localStorage(chat_timer_{user_id})`。

- **当前明显缺口/风险（实验视角）**
  - **condition 分配不一致**：
    - `JsonStorage.create_user()` / `register_user_no_invite()` 会随机分配 `self_disclosure_condition`。
    - `JsonStorage.register_user()`（走邀请码注册）目前**没有**写入 `self_disclosure_condition`，导致该路径注册的用户可能在 `get_condition()` 时回退到默认 `emotional`。
  - **实验流程页缺失**：目前前端只有 `/login /register /chat /settings /admin`。
  - **实验计时可信度**：当前计时完全在前端 localStorage，刷新/多端可能绕过；若要实验严谨，建议引入服务端 session 状态。

## 1. 实验整体流程（目标态）

参考 `reference/实验开始页可视化.html` 的交互 + 你提供的“实验事项”计划，建议将一次实验拆为 4 步（与参考页时间轴一致）：

1. **账号登录**（已存在）
2. **前测问卷**（新增实验开始页）
3. **正式对话**（复用 `/chat`，但由实验入口触发开始；时长 15min 或 16min 以最终方案为准）
4. **后测反馈**（新增实验结束页：展示二维码/链接，引导提交并确认）

> 链接与二维码图片先用占位符：后续你提供真实跳转链接与图片 URL 后再替换。

## 2. 页面与路由规划（前端 Vue3）

### 2.1 新增路由

- `GET /experiment/start`
  - 参考：`reference/实验开始页可视化.html`
  - 功能：展示时间轴 + 右侧二维码卡片 + “是否已提交”确认；确认“是”才允许进入聊天。

- `GET /experiment/end`
  - 功能：对话结束后展示后测二维码/链接，确认提交后可结束或跳转设置页/退出。

> 是否需要把 `/` 默认重定向从 `/chat` 改为 `/experiment/start`：
> - 如果实验模式是默认入口：建议改。
> - 如果实验只是一个模式：建议保留 `/chat`，再提供一个“进入实验”按钮入口。

### 2.2 实验开始页（/experiment/start）关键交互

页面结构建议复刻参考 HTML：

- **左侧时间轴**：
  - Step1 登录：done
  - Step2 前测问卷：active
  - Step3 正式对话：pending（提示 15min）
  - Step4 后测反馈：pending

- **右侧二维码卡片**：
  - 二维码图片：`{{PRETEST_QR_IMAGE_URL}}`
  - 链接：`{{PRETEST_LINK}}`

- **确认区**：
  - 两个选项：
    - 还没完成
    - 是，已提交
  - `Next` 按钮 disabled，只有选“是，已提交”才 enable。
  - 点击 Next：
    - 推荐：先调用后端 `POST /api/experiment/pretest/confirm`（见第 3 节），再跳转 `/chat`。

### 2.3 Chat 页（/chat）在实验模式下的行为

现有 Chat 页已经有计时与结束遮罩：

- **现状**：第一次发消息时启动计时器；15min 提示；16min 强制结束，显示“今日聊天时间已结束”。

实验需要的调整建议（仅在文档先明确，不强制立刻改）：

- **入口控制**：从 `/experiment/start` 进入 `/chat` 后，视为“实验对话开始”。
- **计时开始时机**（二选一）：
  - A（最小改动）：继续沿用“用户第一次发消息启动计时”。
  - B（更符合实验）：进入 `/chat` 即启动计时（不等首次发消息）。

- **结束后的跳转**：
  - 目前是遮罩 + “去设置页面”。
  - 实验建议：到达时长后自动/引导跳转 `/experiment/end`。

### 2.4 实验结束页（/experiment/end）

建议布局与开始页保持一致风格（可以复用同一组件/样式）：

- **右侧二维码卡片**：
  - 二维码图片：`{{POSTTEST_QR_IMAGE_URL}}`
  - 链接：`{{POSTTEST_LINK}}`

- **确认区**：
  - “是否已完成提交？”
  - 选“是”后：
    - `POST /api/experiment/posttest/confirm`
    - 然后显示“感谢参与”或跳转 `/settings` 或直接 `logout`（按实验需求选定）。

## 3. 后端接口与状态模型（建议）

### 3.1 实验状态数据（建议最小字段）

建议在 `users.json` 用户对象里新增/维护（或单独 `experiment_sessions.json` 也可）：

- `experiment_pretest_confirmed_at`: ISO string | null
- `experiment_chat_started_at`: ISO string | null
- `experiment_chat_ended_at`: ISO string | null
- `experiment_posttest_confirmed_at`: ISO string | null

> 目的：
> - 控制用户是否可以进入 chat
> - 控制 chat 是否应结束
> - 保障刷新/多端一致

### 3.2 推荐新增 API（/api/experiment/*）

- `GET /api/experiment/status`
  - 返回：当前用户实验状态 + 本次实验配置（时长、链接占位符/真实链接）。

- `POST /api/experiment/pretest/confirm`
  - 作用：记录用户已完成前测（不校验问卷系统真实性，仅作流程 gate）。

- `POST /api/experiment/chat/start`
  - 作用：记录 chat 开始时间（并可同时 `SessionStateService.reset_session(user_id)` 以确保 turn_count 等从 0 开始）。

- `POST /api/experiment/chat/end`
  - 作用：记录 chat 结束时间。

- `POST /api/experiment/posttest/confirm`
  - 作用：记录后测已提交。

> 注：如果短期不想新增后端模块，也可以先全部走前端 localStorage，但建议至少保留这套接口设计，后续实验严谨化时再落地。

## 4. 实验分组（condition）策略与落地

### 4.1 目标

确保每个用户在实验期间稳定落在三组之一：

- `emotional`（情感表露）
- `factual`（事实表露）
- `none`（无表露）

并且由后端决定，模型只执行对应系统提示词。

### 4.2 建议分配规则（与你的计划一致）

- **确定性映射（推荐）**：基于 `user_id` 做取模：
  - `user_id % 3 == 0` → `none`
  - `user_id % 3 == 1` → `emotional`
  - `user_id % 3 == 2` → `factual`

优点：可复现、便于平衡样本。

### 4.3 需要修复的现状问题

- 目前 `register_user()`（邀请码注册）未写入 `self_disclosure_condition`。
- 目前 `create_user()` / `register_user_no_invite()` 是随机分配。

建议统一为同一策略（随机或取模二选一），并保证三条注册路径一致。

## 5. 与现有对话引擎 v2 的衔接点（清晰化）

一次聊天请求（`/api/chat/send_with_context`）当前后端行为：

- 从用户数据读取 `self_disclosure_condition`
- `PromptManager` 加载对应 prompts
- `SessionStateService.compute_preset()` 计算 preset
- 模型返回 JSON → `ResponseParser` 解析
- 写入 chat_history（按 segments 分条写入）

实验功能主要在“对话前后”的**流程控制**，不需要改动核心对话引擎，只需要：

- 在进入 chat 前：
  - 确认 pretest
  - 重置 session（可选）
  - 记录开始时间

- 在达到时长后：
  - 记录结束时间
  - 引导到 /experiment/end

## 8. 验收标准（Definition of Done）

- 能从登录后进入 `/experiment/start`，看到前测二维码与链接（占位符可正常展示）。
- 用户未确认“已提交”时，无法进入 chat。
- 进入 chat 后能完成 15/16 分钟对话流程。
- 到时后能进入 `/experiment/end`，展示后测二维码与链接（占位符可正常展示）。
- 后端对每个用户稳定记录 `self_disclosure_condition`，且三组提示词可按条件生效。

- **[结束页跳转]**：B - 保留遮罩，加按钮点击后跳转 `/experiment/end`
